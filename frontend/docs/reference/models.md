# 模型与自检

模型列表不是一张固定菜单，而是当前 Key 的实时视图。最稳妥的做法，是用准备接入的那把 Key 直接查询当前真正可用的模型。

::: important 文档里的模型名只能当参考
真正能不能用，取决于这把 Key 当前绑定的分组、该分组的平台类型和它背后的上游调度状态。
:::

## 为什么不要只看文档里的模型名

模型可见性会受这些因素影响：

- Key 绑定的分组
- 分组的平台类型
- 分组下当前可调度账号
- 某些平台自己的模型映射规则

所以最稳妥的方法永远是：**用你准备接入的那把 Key 自己查。**

## Claude / OpenAI 兼容分组怎么查

```bash
curl https://puaai.xyz/v1/models \
  -H "Authorization: Bearer sk-your-api-key"
```

这个结果最适合拿来给：

- OpenAI 兼容客户端
- Claude 兼容客户端
- 需要模型下拉框或自动补全的前端

## Gemini 分组怎么查

```bash
curl https://puaai.xyz/v1beta/models \
  -H "Authorization: Bearer sk-your-api-key"
```

如果你走的是 Gemini SDK / CLI，这个列表比 `/v1/models` 更有参考价值。

## Antigravity 为什么更要先查

Antigravity 分组有几个典型特征：

- 可能存在模型名映射
- 可能参与混合调度
- 某些看起来“像官方”的模型名，实际要以当前分组返回值为准

因此这类分组更不建议手填模型名。

## 自检顺序建议

1. 先请求模型列表
2. 从返回结果里复制模型名
3. 再把模型名填进客户端或代码里
4. 发一个最小消息请求验证

::: tip 最稳妥的接入方式
在模型列表里复制模型名，比自己记忆或照搬旧截图更可靠，尤其是在存在映射、回退或混合调度的平台上。
:::

## 常见误区

### 误区 1：主站显示过某模型，所以这把 Key 一定能用

不一定。你这把 Key 只能看到自己分组允许的模型。

### 误区 2：同一个模型名在所有协议里都能通用

不一定。不同协议兼容层、不同平台分组，对模型名的接受程度可能不同。

### 误区 3：模型列表能返回，就代表任何高级能力都一定可用

不一定。模型可见只是第一步，流式、工具调用、图像、音频、Sora 媒体等能力还要以具体分组和上游支持情况为准。

## 排错建议

如果你遇到“模型不存在”或“请求被拒绝”，按这个顺序查：

1. 当前 Key 是否绑定了正确分组
2. 模型名是不是从当前 Key 的模型列表里复制出来的
3. 你走的入口是不是和平台类型匹配
4. 当前分组是否切到了其他平台或发生了回退 / 映射变化
